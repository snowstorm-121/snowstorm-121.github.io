核心思维框架（贯穿全篇，先放在最前面）

这几天所有的坑，最后都能归结到同一条判断逻辑：

能不能用"沿内存连续顺序读取"描述出你想要的结果？

  • 能 → 直接给你一个"视图"（view），共享同一块内存，不拷贝
  • 不能 → 必须新建内存，拷贝一份数据

这条逻辑解释了：tensor()/from_numpy()/as_tensor() 的拷贝差异、view 为什么对 permute 后的结果报错、基础切片 vs 花式索引/布尔索引的拷贝差异。建议把这条逻辑作为回忆整个章节的钩子。

一、创建 Tensor 的几种方式

1.1 torch.tensor() / torch.as_tensor() / torch.from_numpy()：拷贝行为不同

import torch
import numpy as np

arr = np.array([1.0, 2.0, 3.0])

t1 = torch.tensor(arr)
t2 = torch.as_tensor(arr)
t3 = torch.from_numpy(arr)

arr[0] = 999.0

print(t1)  # tensor([1., 2., 3.])      —— 没变，独立拷贝
print(t2)  # tensor([999., 2., 3.])    —— 变了，共享内存
print(t3)  # tensor([999., 2., 3.])    —— 变了，共享内存

三者的设计意图不同，不是行为随机：

| | | |
|---|---|---|
|函数|语义|拷贝行为|
|torch.tensor()|"我要一个全新独立的tensor"|总是拷贝，无论输入是list还是numpy数组|
|torch.from_numpy()|"我要和这个numpy数组共享内存"|永不拷贝（这是它存在的核心目的）|
|torch.as_tensor()|"能不拷贝就不拷贝"|条件性：dtype兼容且输入是numpy数组（或类似buffer）才共享；如果输入是Python list，则无法共享，行为等同于 tensor()|

补充验证（list场景）： 把 arr 换成 Python list 后，torch.tensor(lst) 和 torch.as_tensor(lst) 都不会共享内存——因为 list 本身在内存里不是连续 buffer，as_tensor 没有"可复用的内存"可借，只能新建。

关于"以后就不用 as_tensor 了"的思考（这是个值得记的判断框架，不只是技术点）：

as_tensor 的行为不是不确定，是有清晰条件的（数据源类型 + dtype是否兼容）。直接弃用等于回避了"我到底要不要让这个tensor和原数据共享内存"这个本该想清楚的问题，换来的"安全感"是假的，而且会主动放弃它能带来的性能优化（避免不必要拷贝）。

更合理的判断方式：

  • 默认场景、不确定原数据后续会不会被改 → 用 tensor()，最安全
  • 明确希望共享内存、为了省内存或性能 → from_numpy() / as_tensor()，主动选择，不是图省事

1.2 dtype 默认值不统一

a = torch.zeros(3)       # dtype: float32
b = torch.arange(3)      # dtype: int64

规律： "填充值"类函数（zeros/ones/full）默认给 float32（因为深度学习场景大多数tensor最终要参与浮点运算）；"生成序列"类函数（arange）如果传入整数参数，保留整数语义，给 int64。

坑点： 容易默认认为所有创建函数都该是浮点，结果用 torch.arange() 生成的索引/序列 tensor 是 int64，后续如果想对它 requires_grad_(True) 会报错——报错信息会指向dtype问题，但第一反应未必会想到"创建的时候就选错了函数"。

主动控制方式：

idx = torch.arange(5, dtype=torch.float32)
idx.requires_grad_(True)  # 现在不报错

建议：凡是涉及 requires_grad 的 tensor，创建时显式写 dtype，不依赖默认值——这也是给自己/给阅读代码的人一个明确信号。

torch.full((2,3), 7) 的 dtype 推断逻辑同 tensor()：看填充值是int还是float（7 → int64，7.0 → float32）。

1.3 为什么整数 dtype 不能 requires_grad

t_int = torch.tensor([1, 2, 3])
t_int.requires_grad_(True)
# RuntimeError: only Tensors of floating point dtype can require gradients

原因（不是框架随意设的限制，是数学概念本身决定的）： 梯度的本质是"输出对输入的变化率"，这个概念要求一个连续可微的空间——你要能问"x变化一点点，y变化多少"。整数是离散的，1和2之间没有"中间状态"，变化率在整数空间没有意义。浮点数是用来近似连续空间的，整数不是。

实战提醒： 如果某个tensor是从整数标签（如分类label）转过来的，想对它求梯度时报错——这其实是在提示"这个东西本不该被求梯度"，单纯换dtype消掉报错而不想清楚是否真的该参与梯度计算，是治标不治本。

二、形状操作

2.1 view vs reshape

  • view()：要求底层内存连续（contiguous），不连续直接报错，不拷贝
  • reshape()：能用view就用view（不拷贝），不能就自动拷贝一份，保证一定成功

坑点：reshape 的"宽容"会掩盖一个重要信号——这次操作有没有发生拷贝。习惯性全用 reshape 会让你失去"如果这里view报错，说明当前内存布局不连续，为什么不连续？"这个反思机会，这个问题往往牵出更深的操作链路问题（比如前面做过transpose或切片）。

a = torch.arange(6)
b = a.view(2, 3)
b[0, 0] = 999
print(a)  # tensor([999, 1, 2, 3, 4, 5]) —— view共享内存，验证过

2.2 stride 概念精讲（本阶段最难啃的一块，多轮才理解透）

stride 是什么： tensor 除了"数据"本身（内存里一条线性排列的数字），还有"怎么读这串数据"的信息，即形状（shape）+ 步长（stride）。stride() 的每个数字表示"沿这一维移动一步，要跳过内存里几个位置"。

用stride计算任意元素在内存中的位置：

位置 = 行号 × stride[0] + 列号 × stride[1] + ...

验证：

a = torch.arange(6).reshape(2, 3)
print(a.stride())  # (3, 1)
# a[1,1] 位置 = 1×3 + 1×1 = 4 → 内存位置4的值是4，对上

"标准款" stride 是什么： 给定一个形状，从数据"直接从左到右顺序切"得到的、唯一自然的排法，对应的stride。规则：最后一维的stride永远是1，往前每一维的stride = 它后面所有维度大小的乘积。

例：形状(2,3)的标准stride是(3,1)；形状(3,2)的标准stride应该是(2,1)。

permute 做了什么： 只重新排列了维度的"读取顺序"（stride的顺序），没有挪动底层任何数据。

a = torch.arange(6).reshape(2, 3)   # 内存: [0,1,2,3,4,5]，stride (3,1)
b = a.permute(1, 0)                  # 形状变(3,2)，stride变(1,3)
print(b)
# tensor([[0, 3],
#         [1, 4],
#         [2, 5]])
print(b.is_contiguous())  # False

b 的实际stride是 (1,3)，最后一维是3，不是"标准款"要求的1——所以不连续。

为什么不能用"自然切法"直接得到b的内容： 如果用"从左到右顺序切"的自然方式去切(3,2)形状，会得到 [[0,1],[2,3],[4,5]]，但 permute 真正想要的是 [[0,3],[1,4],[2,5]]——两者结果不同，不是"自然方式失败"，是自然方式服务的目的（直接平铺）和permute的目的（行列互换）本来就不是一回事，所以方法不可能一样。permute必须保证语义正确（真正的行列互换）同时不挪动底层数据，只能换一套"跳着读"的stride去实现。

为什么 view 对 permute 后的结果报错：view 只会用"标准款"stride去解读数据（只会"顺着读"），它没有"跳着读"的能力。permute之后的stride不是标准款，view无法工作，直接拒绝。

reshape/.contiguous() 怎么解决： 先把数据按当前"看到的样子"真实拷贝、重新摆放成一条新的连续内存，摆好后再用标准stride读就对了——这就是发生的那次隐藏拷贝。所以常看到 .permute(...).contiguous().view(...) 这种链式写法，不是炫技，是没有 .contiguous() 这步会直接报错。

2.3 squeeze / unsqueeze：隐藏维度坑

x = torch.zeros(1, 3, 1)
y = x.squeeze()
print(y.shape)  # torch.Size([3])

坑点： 无参数的 squeeze() 会把所有大小为1的维度一次性去掉，不是只去一个。实战中容易在 batch_size 刚好是1的调试场景（单条样本调试很常见）下，不小心把batch维度也一起去掉，导致下游代码维度数不对，报错可能发生在很远的地方。

安全做法： 永远显式指定 dim，如 x.squeeze(0)——只去掉第0维，如果第0维不是1则什么都不做、不报错。

三、索引切片

3.1 三种索引方式的内存行为差异

| | | | |
|---|---|---|---|
|索引方式|示例|是否共享内存|原因|
|基础切片（冒号）|a[1:4]|共享（视图）|取出的范围在内存里本身连续，可以用"起点+长度"描述|
|花式索引（用tensor/list当索引）|a[torch.tensor([1,3,4])]|不共享（拷贝）|取出的位置在内存里不连续，无法用简单规律描述|
|布尔索引|a[a > 0]|不共享（拷贝）|同花式索引，本质上内部也是转成不连续下标去取|

三组均已动手验证，结果与"能否用连续顺序描述"这条核心逻辑完全吻合。

a = torch.arange(6)
b = a[1:4]
b[0] = 999
print(a)  # tensor([0, 999, 2, 3, 4, 5]) —— 共享内存

a = torch.arange(6)
idx = torch.tensor([1, 3, 4])
c = a[idx]
c[0] = 999
print(a)  # tensor([0, 1, 2, 3, 4, 5]) —— 未变，拷贝

a = torch.tensor([1, -2, 3, -4, 5])
mask = a > 0
b = a[mask]
b[0] = 999
print(a)  # tensor([1, -2, 3, -4, 5]) —— 未变，拷贝

3.2 省略号 ...

... 表示"中间这些维度全部保留，不用一个个写冒号"，纯粹是简写，没有额外坑。

x = torch.arange(24).reshape(2, 3, 4)
print(x[..., 0].shape)   # torch.Size([2, 3])  等价于 x[:, :, 0]
print(x[0, ...].shape)   # torch.Size([3, 4])  等价于 x[0, :, :]

维度多的场景（如图像数据 (batch, channel, height, width)）能省去写多个冒号。

四、广播机制（Broadcasting）

规则： 从最后一维开始往前比较，每一维需满足以下任一条件，否则报错：

1. 两边这一维大小相等
2. 其中一边这一维大小是1
3. 其中一边没有这一维（维度数不够，缺的当1处理，从左侧补）

最危险的坑：不是"报错"，是"以为会报错、结果悄悄广播成功了，但语义不是你想要的"

x = torch.zeros(3, 1)
y = torch.zeros(3)
z = x + y
print(z.shape)  # torch.Size([3, 3])  —— 不报错，但结果可能不是预期

推导过程：

  • x: (3,1)，y: (3,)
  • 维度数对齐（y前面补1）：x: (3,1)，y: (1,3)
  • 最后一维：x是1，y是3 → 广播成3
  • 前一维：x是3，y是1（刚补的） → 广播成3
  • 结果：(3,3)

实际发生的事：x 的列被横向复制3次，y 的行被纵向复制3次，对应位置相加，得到一个3×3的表格，每个位置是"x第i个数 + y第j个数"。

实战提醒： 如果某个tensor算出来形状不小心多了一个大小为1的维度（如(N,1)本该是(N,)），跟另一个(N,)的tensor做运算，不会报错，会悄悄得到(N,N)的结果。这种bug往往要到后面（比如loss算出离谱的数字）才会被发现，而且很难联想到"罪魁祸首是几步之前一个多余的维度"。这条直接呼应"随时清楚自己tensor的形状"这个习惯。

五、原地操作（In-place Operations）

PyTorch中带尾部下划线的函数（add_、mul_、sub_、requires_grad_等）都是直接在原tensor上修改，不新建tensor，这是统一的命名约定。

a = torch.tensor([1.0, 2.0, 3.0])
a.add_(1)
print(a)  # tensor([2., 3., 4.])  —— a自己被改了

坑点：原地操作会绕过"新建拷贝"这个安全网，与共享内存叠加时容易制造"远处代码改变了变量"的诡异bug

a = torch.tensor([1.0, 2.0, 3.0])
b = a.view(3)      # b 和 a 共享内存
b.add_(10)
print(a)  # tensor([11., 12., 13.])  —— a 没被任何代码直接赋值，但变了

如果这种代码埋在几十行的训练循环里，排查时很容易卡住，因为表面上从没写过 a = ... 这种赋值语句去改它，违反"我没写代码改这个变量，它就不该变"的直觉。

该养成的习惯： 用原地操作前先问自己——这个tensor现在是不是和别的变量共享内存？如果是，改的其实是"所有指向这块内存的变量"。判断是否共享内存的方法，就是本篇第一部分到第三部分一路验证下来的那条逻辑。

埋个钩子（留给 Autograd 阶段）： 对需要计算梯度的tensor做原地操作，PyTorch会有更严格的限制（很多情况直接报错）。原因：反向传播需要记得tensor"原来的样子"，原地操作把原来的样子覆盖掉了，会让梯度计算的依据丢失。具体机制留到正式学Autograd时展开。

六、Tensor 合并与拆分：cat / stack / split / chunk

6.1 cat vs stack：合并时是否新增维度

a = torch.tensor([1, 2, 3])
b = torch.tensor([4, 5, 6])

c1 = torch.cat([a, b], dim=0)
c2 = torch.stack([a, b], dim=0)

print(c1.shape, c1)  # torch.Size([6])    tensor([1,2,3,4,5,6])
print(c2.shape, c2)  # torch.Size([2,3])  tensor([[1,2,3],[4,5,6]])
  • cat：在已有维度上拼接，不新增维度，原来几维还是几维，只是某一维变长
  • stack：新增一个维度再堆叠，多出的维度用来区分"这是第几个原始tensor"

实战判断： 多张形状相同的单张图片要拼成batch → 用stack（每张图片要保留独立性，多一个batch维度）；已经是batch维度的多段数据想首尾接长 → 用cat。

6.2 split vs chunk：按"每块多大" vs "分成几块"

x = torch.arange(10)

s1 = torch.split(x, 3)
# (tensor([0,1,2]), tensor([3,4,5]), tensor([6,7,8]), tensor([9]))

s2 = torch.chunk(x, 3)
# (tensor([0,1,2,3]), tensor([4,5,6,7]), tensor([8,9]))
  • split(x, n)：严格按"每块n个"切，块数不固定，由总数和每块大小决定，最后一块可能远小于n
  • chunk(x, n)：你指定"分成n块"，尽量平均分，块数固定，大小不一定完全相等（除不尽时前面块稍大）

坑点： 容易下意识假设"chunk出来的每一块大小一定相等"（"chunk"听起来像均匀切块）。遇到不能整除的情况，最后一块会比前面小，如果后续代码假设每块都能reshape成固定形状，会在边界条件报错或得到错误结果。

八、面试高频追问点小结

整理这几天最容易被追问"为什么"的几个点，复习时可以用来自测：

1. torch.tensor() / as_tensor() / from_numpy() 拷贝行为为什么不同？—— 三者设计意图不同，不是随机行为
2. 为什么整数dtype不能 requires_grad？—— 导数要求连续可微空间，整数是离散的
3. view 和 reshape 的本质区别是什么？—— view要求stride是"标准款"（对应连续内存的顺序读取），reshape在做不到时自动拷贝兜底
4. 什么是"标准款"stride，permute之后为什么不是？—— 标准款是"从内存顺序直接切出形状"对应的唯一stride；permute只改变了维度的逻辑顺序，没挪动底层数据，导致新的stride不再对应连续内存布局
5. 基础切片、花式索引、布尔索引，哪些共享内存哪些不共享，为什么？—— 取决于目标位置能否用"连续顺序"描述
6. 广播规则的判断条件是什么？最容易踩的坑是什么？—— 从最后一维往前比，相等/其中一边是1/其中一边维度不存在；最危险的坑是"以为会报错结果却悄悄成功了，语义不对"
7. 原地操作（_结尾）的风险点是什么？—— 与共享内存叠加时，会在"看不出哪里改的"的地方修改变量值
